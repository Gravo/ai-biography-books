# 《Ashish Vaswani 传记：Transformer 架构之父》

## Chapter 5：Adept AI的技术架构——ACTION模型

> **核心论点**：Adept AI的"ACTION模型"不是"更大的LLM"——它是**多模态AI代理**（Multimodal AI Agent）。核心创新是：**让Transformer"看到"屏幕 + "操作"鼠标键盘**。这比ChatGPT难10倍（需要理解UI、规划多步任务、处理失败）。

## 🖥️ 让AI"操作电脑"——为什么难？

### 任务示例

```
任务："帮我预订明天去纽约的航班"

**人类做法**：
  1. 打开浏览器
  2. 进入航空公司网站
  3. 填写"出发地/目的地/日期"
  4. 点击"搜索"
  5. 选择航班
  6. 填写乘客信息
  7. 付款
  8. 确认

**ChatGPT做法**：
  → 输出文本："您可以访问航空公司网站，填写..."
  → **不能**实际"打开浏览器"或"点击按钮"

**Adept的ACTION模型做法**：
  → 实际打开浏览器 ✅
  → 实际点击按钮 ✅
  → 实际填写表单 ✅
  → 完成后通知用户 ✅
```

### 技术挑战

```
挑战1：**理解UI**（用户界面）

  问题：
    → 屏幕是"像素矩阵"（224×224×3的图像）
    → AI需要"理解"哪些是按钮、哪些是文本框、哪些是下拉菜单

  解决方案：
    → **Vision Transformer (ViT)**：把屏幕截图当成"图像"处理
    → 或者**UI解析**：用OCR + 目标检测识别UI元素

挑战2：**规划多步任务**

  问题：
    → "预订航班"需要**7-10步**（如上例）
    → 每一步都依赖前一步的结果（"搜索"后才能"选择航班"）
    → LLM擅长"单步生成"，不擅长"多步规划"

  解决方案：
    → **Hierarchical Planning**（分层规划）
    → 高层规划器：决定"下一步做什么"（打开浏览器 → 进入网站 → ...）
    → 底层执行器：决定"如何做"（点击哪个坐标 → 输入什么文本）

挑战3：**处理失败**

  问题：
    → 网站加载失败 → 怎么办？
    → 按钮位置变了 → 怎么办？
    → 需要"验证码" → 怎么办？

  解决方案：
    → **Reinforcement Learning (RL)** 训练"错误恢复"策略
    → 或者**Human-in-the-Loop**：遇到困难时请求人类帮助
```

## 🧠 ACTION模型——可能的架构

### 推测架构（基于公开信息）

```
ACTION模型 = Vision Encoder + Language Model + Action Decoder

输入：
  → **屏幕截图**（图像：224×224×3）
  → **任务描述**（文本："帮我预订明天去纽约的航班"）

输出：
  → **动作序列**（JSON格式）：
    [
      {"action": "open_browser", "url": "https://www.united.com"},
      {"action": "click", "element": "departure_city_input"},
      {"action": "type", "text": "San Francisco"},
      {"action": "click", "element": "search_button"},
      ...
    ]

架构细节（推测）：

  1. **Vision Encoder**（视觉编码器）：
     → 可能是 **ViT (Vision Transformer)** 或 **CLIP** 的变体
     → 输入：屏幕截图（图像）
     → 输出：图像表示向量（shape: [1, d_model]）

  2. **Language Model**（语言模型）：
     → 可能是 **Transformer Decoder**（类似GPT）
     → 输入：任务描述（文本） + 图像表示向量
     → 输出：隐藏状态（shape: [seq_len, d_model]）

  3. **Action Decoder**（动作解码器）：
     → 可能是 **MLP**（多层感知机）
     → 输入：隐藏状态
     → 输出：动作类型（open_browser / click / type / ...） + 动作参数（url / element / text / ...）
```

### 训练数据——如何收集？

```
挑战：**没有"AI操作电脑"的大规模数据集**

解决方案1：**模仿学习 (Imitation Learning)**

  做法：
    → 让人类"演示"如何完成任务（同时记录屏幕 + 鼠标/键盘操作）
    → AI学习"看到这个屏幕 → 应该做这个动作"

  问题：
    → 需要**大量**人类演示（每个任务需要100-1000次演示）
    → 成本高（人类时间）

解决方案2：**强化学习 (Reinforcement Learning)**

  做法：
    → AI自己尝试（可能做错）
    → 如果完成任务 → 奖励 +1
    → 如果失败 → 奖励 -1
    → 通过试错学习

  问题：
    → 搜索空间巨大（"点击哪里？"有数百万像素）
    → 训练不稳定（可能需要数百万次尝试）

解决方案3：**合成数据 (Synthetic Data)**

  做法：
    → 用**程序**自动生成"任务-演示"对
    → 例如：随机生成网站 + 随机生成"填写表单"任务

  优势：
    → 可以生成**无限**数据
    → 成本低（不需要人类）

  问题：
    → 合成数据 ≠ 真实数据（"模拟器偏差"）
    → AI在合成数据上训练，可能在真实网站上失败
```

## 🔬 Adept的技术路线——与OpenAI对比

### OpenAI路线："更大的LLM"

```
OpenAI的假设：
  → "Scale is All You Need"
  → 只要模型足够大、数据足够多 → 就能做**任何事**

证据：
  → GPT-3（2020）：175B参数，Few-shot学习
  → GPT-4（2023）：多模态（文本 + 图像），推理能力更强
  → O1（2024？）："推理时计算"（Chain-of-Thought）

优势：
  → 通用性强（一个模型做所有事）
  → 迁移学习能力强（预训练 → 微调 → 新任务）

劣势：
  → **不能**实际"操作电脑"（只能生成文本）
  → 需要"工具使用"（Tool Use）才能"行动"
```

### Adept路线："专门化的AI代理"

```
Adept的假设：
  → "通用LLM**不够**——我们需要**专门化**的AI代理"
  → "理解语言" ≠ "操作电脑"
  → 需要**新的架构**（ACTION模型）

证据：
  → Adept的招聘信息（2022-2023）：
    - "训练模型操作网页界面"
    - "多模态学习（文本 + 图像 + 动作）"
    - "强化学习 for UI导航"

优势：
  → **专门化**（专注于"操作电脑"）
  → 可能比GPT-4**更擅长**多步任务

劣势：
  → 泛化能力可能不如GPT-4（只擅长"操作电脑"，不擅长"写诗"）
  → 技术风险高（如果ACTION模型work不如GPT-4的"工具使用"）
```

## 💡 技术洞察：为什么"A操作电脑"比"生成文本"难？

### 对比矩阵

```
任务特征                "生成文本"（ChatGPT）   "操作电脑"（Adept）
───────────────────  ───────────────────   ───────────────────
输入                  文本                      文本 + 图像（屏幕截图）
输出                  文本                      动作序列（JSON）
错误代价              低（重新生成即可）        高（可能删除文件、误付款）
反馈延迟              秒级                      分钟级（完成多步任务）
搜索空间              词表大小（50,000词）     屏幕像素（1920×1080 = 2M可能点击位置）
泛化需求              中（新任务可以Few-shot）   高（新网站需要"理解UI"）
可解释性              中（可以看到生成的文本）    低（"为什么点击这里？"难回答）
```

### 关键难点："理解UI"

```
问题：
  → 网站A的"搜索按钮"可能是**蓝色矩形**
  → 网站B的"搜索按钮"可能是**放大镜图标**
  → 网站C的"搜索按钮"可能是**文字链接**

人类可以"泛化"：
  → "哦，这个蓝色矩形是按钮"
  → "哦，这个放大镜图标是按钮"
  → "哦，这个文字链接是按钮"

AI需要学习：
  → **视觉特征**（颜色、形状、纹理）
  → **语义特征**（"搜索"这个词的语义）
  → **上下文特征**（按钮通常在输入框旁边）

这可能需要的架构：
  → **多模态融合**（视觉特征 + 语义特征 + 上下文特征）
  → 比"纯文本LLM"复杂10倍
```
